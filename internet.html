How the Internet Works

Understand what happens behind the scenes when you type a URL into your browser — from IP addresses to HTTP requests and web servers.

01

What is the Internet?

The Internet is a global computer network designed for storing, processing, and transmitting information. As of 2021, nearly 60% of the world's population are internet users — making web-related skills more in demand than ever.

Think of the Internet as a massive system of connected computers, constantly exchanging data with each other using agreed-upon rules.

02

Clients and servers

Every connection on the Internet involves two types of computers:

🖥 Server

A dedicated computer always connected to the Internet. It stores files and content, and sends them to whoever asks.

💻 Client

Your computer, phone, or browser. It requests content from a server and displays it on your screen.

When you visit a website, your browser (the client) sends a request to a server. The server finds the requested file and sends it back. The file is not saved on your computer — it is displayed temporarily while you view it. This is why you always see the latest version of a page.

03

IP addresses

Every computer connected to the Internet has a unique IP address — like a postal address for your device. It allows computers to find and communicate with each other.

IPv4 — the most common format

IPv4 addresses are 4 bytes long, written as four numbers (0–255) separated by dots:

Example
91.198.174.192   ← This is the IP address of Wikipedia

IPv6 — the newer format

Because IPv4 addresses are running out (only ~4 billion possible), IPv6 was introduced. It is 16 bytes long and can support vastly more devices:

Example
1050:0:0:0:5:600:300c:326b   ← IPv6 address
Analogy: An IP address is like a phone number. Every device has a unique one so it can be reached on the network.
04

Domain names

IP addresses are hard for humans to remember. Domain names are human-friendly labels that map to an IP address — like a contact name in your phone replacing a long number.

A system called DNS (Domain Name System) translates domain names into IP addresses behind the scenes.

Example
# Both of these take you to the same website:
http://91.198.174.192       ← IP address (hard to remember)
http://wikipedia.org        ← Domain name (easy to remember)

The domain name system is organized in levels separated by dots. For example, in en.wikipedia.org:

  • .org — top-level domain (TLD), indicates the type of organisation
  • wikipedia — the second-level domain (the site's name)
  • en — a subdomain (the English-language version)
05

Protocols

A protocol is a set of rules that defines how computers exchange data. Different types of data use different protocols.

ProtocolFull nameUsed for
HTTP / HTTPSHyperText Transfer ProtocolExchanging web pages (HTML documents)
FTPFile Transfer ProtocolTransferring files between computers
SMTPSimple Mail Transfer ProtocolSending emails
IMAPInternet Message Access ProtocolReceiving and reading emails
IRC / IoTInternet Relay Chat / Internet of ThingsBuilding connected devices
Analogy: Protocols are like languages. If two computers want to exchange data, they must both speak the same protocol — just like two people needing a shared language to communicate.
06

HTTP vs HTTPS

Both HTTP and HTTPS are protocols for transferring web pages. The key difference is security.

HTTP

HyperText Transfer Protocol
Data is sent as plain text. Anyone intercepting the connection can read it. Not secure for sensitive information.

HTTPS 🔒

HTTP Secure
Data is encrypted before it is sent. Even if someone intercepts it, they cannot read it. Used for login pages, payments, and all modern websites.

Example
http://example.com   ← Not secure (avoid for sensitive data)
https://example.com  ← Secure and encrypted ✓
Best practice: Always use HTTPS for any website you build. Browsers show a padlock icon 🔒 for HTTPS sites, and may warn users that HTTP sites are "Not Secure". GitHub Pages gives you HTTPS for free.

Static sites

A static site always returns the same pre-written files from the server, regardless of who is requesting them. The files are ready-made HTML, CSS, and images stored on the server.

  1. The browser sends an HTTP request with the URL of the page.
  2. The server finds the matching file in its file system.
  3. The server sends back an HTTP response containing the file and a status code.
  4. The browser reads the HTML and displays the page.
GitHub Pages hosts static sites. Your index.html and internet.html files are exactly this — ready-made HTML files served directly to visitors.

A dynamic site generates its content on the fly, based on who is requesting it and what data is available. The page is built differently for each visitor.

11

Status codes

Every HTTP response from a server includes a status code — a three-digit number that tells the browser whether the request succeeded or failed.

CodeMeaningWhen you see it
200 OK Success Every normal page load that works correctly
301 Moved Redirect The page has permanently moved to a new URL
404 Not Found File missing The requested page does not exist on the server
500 Server Error Server problem Something went wrong on the server side
What happens when you visit a page
# Page exists → server responds with:
200 OK — here is your HTML file.

# Page does not exist → server responds with:
404 Not Found — no file at that path.

# Something crashed on the server → responds with:
500 Internal Server Error
Tip: On GitHub Pages, if you visit a URL that does not match a file, you will see a 404 error. You can create a file named 404.html in your repo to customise that error page.
